Factory Stages – Internal Process (How Work Moves)

Same flow for all clients: Intake → Formation → Build → Handover. The purchased package only changes which modules activate in Build. There is one client handover only: at the end.

1) Intake & Build Map

Decide what gets built. Freeze scope.

2) Formation (Foundation)

Register the company + setup accounting.

3) Build (Execution)

Build only active modules (override logic).

4) Handover (Exit)

Deliver once. Activate accounting. Close.

1) Intake & Build Map

Gate 0

Purpose: Translate the purchase into a frozen Build Map. No work starts before scope is locked.

Inputs

  • Purchased package (Stage 1 / 2 / 3)
  • Company type (EF / AB) + business model
  • Complexity flags (payroll, EU sales, integrations)

Process (Steps)

  • Confirm purchased stage (locked)
  • Collect structured answers (no free-form)
  • Generate Build Map: Modules ON / OFF / UPGRADED
  • Internal validation (no creativity, only correctness)

Not allowed

  • No execution / no design / no registration
  • No strategy calls or custom consulting
  • No changing modules without new order
Exit Criteria: Build Map approved & frozen → move to Formation.

2) Formation (Foundation)

Backend Only

Purpose: Make the company legally alive + create accounting infrastructure. Client does not receive partial outputs.

Inputs

  • Approved Build Map
  • Client identity details + legal docs

Process (Steps)

  • Company registration (EF or AB)
  • Tax setup: F-skatt + VAT (if applicable)
  • Accounting system setup: chart of accounts + invoice templates
  • Internal verification: legal status + system access

Not allowed

  • No branding, websites, marketing, content
  • No client delivery / no partial handover
  • No skipping Formation
Exit Criteria: Company registered + accounting system live → move to Build.

3) Build (Execution)

Modules Only

Purpose: Execute only the modules required for the purchased stage. No duplication. No extras.

Inputs

  • Approved Build Map (modules ON/OFF/UPGRADED)
  • Active company from Formation

Process (Steps)

  • Activate modules according to Build Map
  • Build per module (brand / web / social / legal / tech)
  • Internal QA: match Build Map + stage outputs

Not allowed

  • No client previews or feedback loops
  • No “nice-to-have” additions
  • No stacking: higher stages replace lower versions
Exit Criteria: All active modules complete + QA passed → move to Handover.

4) Handover (Exit)

Only Client Delivery

Purpose: Deliver once, confirm accounting subscription, close production. Anything new becomes a new order.

Inputs

  • Completed Build + QA passed
  • Accounting plan ready for activation

Process (Steps)

  • Final verification vs Build Map
  • Access transfer (web/admin/accounting)
  • Accounting lock-in (subscription + autopay)
  • Formal close (delivery confirmed, ticket closed)

Not allowed

  • No extra work during handover
  • No scope changes
  • No promises for future tasks without new order
Exit Criteria: Client access confirmed + accounting active → company exits the factory.

Carved in Stone Rules

Always
  • Follow the Build Map. If it’s not listed, it’s not included.
  • One delivery per production cycle: deliver at Handover only.
  • Formation is backend-only and never delivered partially.
  • Modules override (replace). They never stack or duplicate.
Never
  • No partial handovers or mid-production client reviews.
  • No exceptions inside production. Log exceptions; fix system later.
  • No custom scope sold inside packages. New needs = new order.
  • No improvisation during Build. Execution follows templates.
