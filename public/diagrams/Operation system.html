Operational Module (Ops + Delivery)

Operations protects production. Nana owns delivery and scope control. Ayham owns flow, discipline, KPIs, and exceptions. Build never talks to clients directly.

Operational Flow (From “Build Ready” to “Closed”)

This is the factory buffer layer: absorbs client noise, prevents scope creep, and closes projects cleanly.

Step 1 — Build Marks “READY”

Trigger

Ammar confirms QA passed and sends “Ready for delivery” to Nana.

Input: Final website + brand pack + access
Owner: Ammar (QA) → Nana (receives)

Step 2 — Delivery Scheduling

Nana

Nana schedules a delivery call and sends “What you’ll receive” summary.

Rule: Delivery happens in a fixed window
Output: Client calm + expectations set

Step 3 — Controlled Walkthrough

Nana

Nana delivers assets, explains access, confirms scope is complete.

Deliver: links + credentials + files
Confirm: “Delivered” status in system

Step 4 — Scope Protection

Non-Negotiable

Any “extra request” becomes a new order. No debates. No emotional exceptions.

Script: “Outside package → new order.”
Protects: time, margins, team focus

Step 5 — Project Closure

Close Loop

Nana closes the project formally so nothing “hangs” in the background.

Client: acknowledges delivery
Internal: project marked CLOSED

Step 6 — Stabilize Account (Recurring)

Revenue Engine

Client transitions into Accounting and/or Tech Support — or fully closed.

Path A: Accounting subscription
Path B: Tech support subscription
Operational Laws: Build never talks to clients · Nana filters all feedback · Scope changes = new order · Exceptions are logged first (not acted on emotionally) · Closure is mandatory.

Role Ownership

NANA • Head of Operations

Delivery Commander

Protects build, controls scope, and closes loops. Client communication lives here after build starts.

  • Owns delivery scheduling + walkthrough Ensures clients receive everything clearly and on time.
  • Scope control (no exceptions) Extra requests become new orders — always.
  • Project closure + account stabilization Moves clients to accounting/tech support or closes the file.
AYHAM • CEO / Factory Manager

Flow & Discipline Owner

Manages the factory through KPIs, bottlenecks, and system improvements — not by doing the work.

  • Owns KPIs across ALL stages Intake, Foundation, Build, Ops — monitors health and leaks.
  • Owns exceptions + system changes Exceptions are logged first → decided later.
  • Owns discipline & bottleneck removal Fixes flow issues, not individual tasks.
CEO rule: Ayham manages flow, not execution. If he starts doing tasks, the factory will collapse back into chaos.
